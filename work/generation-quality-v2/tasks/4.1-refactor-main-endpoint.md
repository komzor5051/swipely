# Task 4.1: Refactor main generation endpoint to use pipeline

## Description

Replace inline prompt building and Gemini call in `/api/generate/route.ts` with call to `generateCarousel()` from the shared pipeline. Keep all auth, validation, rate limiting, DB save logic. Add `framework` field to request body with enum validation. Raise timeout to 60s.

## Why

This is where the multi-agent pipeline integrates with the existing HTTP endpoint. The endpoint retains all business logic (auth, billing, rate limiting) but delegates content generation to the pipeline (D2, D6, D12).

## Acceptance Criteria

- [ ] `route.ts` calls `generateCarousel()` from `lib/generation/pipeline.ts` instead of inline Gemini call
- [ ] All inline `designPresets`, `contentTones`, `buildSystemPrompt`, `buildSlideStructure`, `buildPreservePrompt` removed (imported from shared module)
- [ ] `framework` field parsed from request body with server-side enum validation against `FrameworkId` values (D12) — invalid values rejected with 400
- [ ] Preserve-text mode passes `preserveText: true` to pipeline (formatter-only path)
- [ ] Timeout raised from 55s to 60s
- [ ] Auth, email verification, profile lookup, rate limiting, slot claiming — all unchanged
- [ ] DB save logic unchanged (saves `output_json` from pipeline response)
- [ ] Error responses from pipeline mapped to appropriate HTTP status codes (500 for agent failure, 502 for Gemini errors)
- [ ] `npm run build` passes

## Verify-smoke

`curl -X POST http://localhost:3000/api/generate` with auth cookie and test payload including `framework: "mistakes"`, verify valid carousel response.

## Verify-user

Generate a carousel on localhost with and without framework selection, verify slides render correctly in both cases.
