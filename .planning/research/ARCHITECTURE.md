# Architecture Patterns: Milestone 2 Integration

**Domain:** Instagram carousel generator SaaS (swipely-nextjs)
**Researched:** 2026-03-19
**Scope:** Four feature areas — layout system, rich elements, photo upload, content calendar

---

## Existing Architecture Baseline

Before describing changes, the relevant existing contracts:

**SlideData (current `components/slides/types.ts`):**
```typescript
export interface SlideData {
  type: string;        // "hook" | "tension" | "value" | "accent" | "insight" | "cta"
  title: string;       // may contain <hl>keyword</hl> tags
  content: string;     // plain text, 20-35 words
  imageUrl?: string;   // base64 data URL — Photo Mode (Gemini-generated images)
}
```

**SlideProps (passed to every template component):**
```typescript
export interface SlideProps {
  slide: SlideData;
  slideNumber: number;
  totalSlides: number;
  format: "square" | "portrait" | "story";
  username?: string;
}
```

**Generation API output (what `/api/generate` returns):**
```json
{
  "slides": [
    { "type": "hook", "title": "Title with <hl>word</hl>", "content": "20-35 word text" }
  ],
  "post_caption": "Caption text"
}
```

**Template rendering chain:**
`SlideRenderer.tsx` → receives `template` prop → looks up component in `TEMPLATE_MAP` → passes `SlideProps` through → component renders inline styles at 1080px canvas

**Layout variant system (already exists in `utils.tsx`):**
`getLayoutVariant(slideType, slideNumber, totalSlides)` already returns `"hero" | "centered" | "split" | "quote" | "cta"` — but this only controls vertical content alignment within a fixed layout. It is not the same as the visual layout variants needed for Milestone 2.

## 1. Layout System

### Problem

All templates currently use a single visual composition: background decorations, logo top-left, content in the center vertical strip, footer bottom. `getLayoutVariant` only changes `justifyContent` (flex-start/center/flex-end). There is no structural layout variation.

### Extended SlideData Schema

Add a `layout` field to `SlideData`. This field is generated by Gemini and consumed by template components:

```typescript
export interface SlideData {
  type: string;
  title: string;
  content: string;
  imageUrl?: string;
  // NEW — Milestone 2
  layout?: "default" | "text-left" | "text-right" | "big-number" | "quote-center" | "split";
  elements?: SlideElement[];   // see Section 2
}
```

`layout` is optional with `"default"` as fallback. Existing slides (no layout field) continue to render exactly as before. This is backward-compatible.

### How Templates Consume layout

Each template component receives `slide.layout` and branches its render output. The approach is a switch inside the component render body — not separate components per layout:

```typescript
// Inside SwipelySlide.tsx (example pattern)
const layout = slide.layout ?? "default";

if (layout === "big-number") {
  // render: giant number centered, small label below
}
if (layout === "quote-center") {
  // render: quote marks, centered text, no title
}
if (layout === "split") {
  // render: left column text, right column image or accent block
}
// else: default layout (existing render path, unchanged)
```

This approach keeps the template file as the single source for one visual identity. The alternative — separate components like `SwipelySlideQuote.tsx` — would duplicate all the decorative layer code (glows, grid, decorations) for each layout variant and create maintenance debt.

### Layout Variants and Their Structural Meaning

| layout value | Visual structure | When Gemini should use it |
|---|---|---|
| `default` | Title top, content below, all full-width | General slides |
| `text-left` | Text in left 55%, right side open for accent/whitespace | value slides with short content |
| `text-right` | Mirror of text-left | visual alternation |
| `big-number` | Giant number (from `elements[0]`) as focal point, small label | accent/stat slides |
| `quote-center` | Large quote marks, centered text block, attribution optional | insight/proof slides |
| `split` | Two equal columns, each with independent content block | contrast/tension slides |

### Prompt Changes in /api/generate

`buildSystemPrompt` must instruct Gemini to include `layout` in each slide:

```
// Add to system prompt rules section:
- layout: один из "default" | "text-left" | "text-right" | "big-number" | "quote-center" | "split"
  - big-number — обязательно для слайдов с числами/статистикой (type: accent)
  - quote-center — для инсайтов и цитат (type: insight, proof)
  - split — для контрастов и напряжения (type: tension, contrast)
  - text-left / text-right — чередуй для value-слайдов
  - default — всё остальное
```

The JSON example in the prompt must also include the `layout` field.

### Validation in /api/generate

Add layout to the slide cleaning step:

```typescript
const VALID_LAYOUTS = ["default", "text-left", "text-right", "big-number", "quote-center", "split"];

carouselData.slides = carouselData.slides.map((slide) => ({
  ...slide,
  title: cleanTitle(slide.title),
  content: preserveText ? slide.content : cleanMarkdown(slide.content),
  layout: VALID_LAYOUTS.includes(slide.layout) ? slide.layout : "default",
}));
```

### Which Templates Should Implement layout Variants

Not all 23 templates need all variants. Start with the 4-5 most-used templates. The architectural contract is the same for all — they just need to handle it gracefully. Unknown layouts fall back to `"default"` rendering.

## 2. Rich Slide Elements

### Extended Schema

Add `elements` array to `SlideData`. Elements supplement (or replace) the `content` string for rich slides:

```typescript
export type SlideElement =
  | { type: "list"; items: string[] }
  | { type: "stat"; value: string; label: string }
  | { type: "progress"; label: string; percent: number }
  | { type: "bar-chart"; data: { label: string; value: number }[] }
  | { type: "pie-chart"; data: { label: string; value: number; color?: string }[] };
```

### Rendering Decision: SVG inline, not Canvas libraries

html2canvas cannot capture Canvas-based chart libraries (Chart.js, Recharts canvas renderer) reliably. SVG renders correctly with html2canvas because it is part of the DOM.

All chart elements must be pure React + SVG. No external chart library dependency.

**Bar chart implementation pattern:**
```typescript
// Pure SVG bar chart — renders in html2canvas
function BarChart({ data }: { data: { label: string; value: number }[] }) {
  const max = Math.max(...data.map(d => d.value));
  return (
    <svg width="900" height="300" viewBox="0 0 900 300">
      {data.map((d, i) => {
        const barHeight = (d.value / max) * 240;
        return (
          <g key={i} transform={`translate(${i * (900 / data.length)}, 0)`}>
            <rect
              x={10} y={300 - barHeight - 40}
              width={900 / data.length - 20} height={barHeight}
              fill="#D4F542"
            />
            <text x={900 / data.length / 2} y={290} textAnchor="middle"
              fill="white" fontSize={20}>
              {d.label}
            </text>
          </g>
        );
      })}
    </svg>
  );
}
```

**Pie chart implementation pattern:**
Compute arc paths via trigonometry in JSX. No d3, no Recharts. ~30 lines of math.

### Where Element Rendering Lives

Create `components/slides/elements/` directory with one file per element type:
- `ListElement.tsx`
- `StatElement.tsx`
- `ProgressElement.tsx`
- `BarChartElement.tsx`
- `PieChartElement.tsx`

Each template component decides how to incorporate elements into its layout. A template that does not support rich elements simply ignores `slide.elements` — backward-compatible.

### Template Rendering Cascade

A slide can have both `content` (text) and `elements`. The template decides how to use both. Recommended rule: if `elements` is non-empty and the layout is `big-number` or `split`, elements take the focal area and `content` serves as caption/subtitle. In `default` layout, elements render below the `content` paragraph.

### Prompt Changes

Gemini must know the elements schema. Add to system prompt:

```
- elements (optional): массив богатых элементов для слайда
  - { type: "list", items: ["пункт 1", "пункт 2", "пункт 3"] } — для перечислений
  - { type: "stat", value: "47%", label: "компаний теряют клиентов" } — для статистики
  - { type: "progress", label: "Конверсия", percent: 73 } — для прогресс-баров
  - { type: "bar-chart", data: [{label: "Янв", value: 45}, ...] } — для столбчатых диаграмм
  - { type: "pie-chart", data: [{label: "Мобайл", value: 70}, ...] } — для пирогов
  Используй elements вместо content когда слайд содержит структурированные данные.
  Максимум 1 chart-элемент на слайд. Список — максимум 5 пунктов.
```

### Validation

In `/api/generate` slide-cleaning step, validate `elements`:
- Check `type` is one of the known values
- Strip unknown element types silently (don't fail the whole generation)
- Cap list items at 7, chart data points at 10

## 3. Photo Upload

### Data Flow

```
User selects file in browser
  → client-side: resize/compress to max 1920px, convert to webp if >500KB
  → POST /api/photos/upload (multipart/form-data)
    → auth check (server Supabase client)
    → upload to Supabase Storage: bucket "user-photos", path "{userId}/{uuid}.webp"
    → return { publicUrl, storageKey }
  → frontend stores { publicUrl, storageKey } in generation state (Zustand)
  → user assigns photo to specific slide(s) via slide editor UI
  → on generate/save: slide JSON includes { imageUrl: publicUrl } for assigned slides
  → SlideRenderer receives imageUrl → template renders PhotoBackground component
```

### Supabase Storage Setup

Bucket: `user-photos`
Path pattern: `{userId}/{uuid}.webp`
Access: Private bucket + signed URLs, or public bucket with RLS restriction on upload path.

Recommended: public bucket with storage policy that restricts upload to `{userId}/*` (users can only write to their own prefix). Signed URLs are not needed for display — public CDN URLs work fine for a carousel generator where images are displayed in-app and exported to PNG.

**Storage RLS policy (SQL):**
```sql
-- Allow users to upload only to their own prefix
CREATE POLICY "Users upload own photos"
ON storage.objects FOR INSERT
TO authenticated
WITH CHECK (bucket_id = 'user-photos' AND (storage.foldername(name))[1] = auth.uid()::text);

-- Allow users to read own photos
CREATE POLICY "Users read own photos"
ON storage.objects FOR SELECT
TO authenticated
USING (bucket_id = 'user-photos' AND (storage.foldername(name))[1] = auth.uid()::text);
```

### New API Route

`POST /api/photos/upload` — handles upload:

```typescript
// app/api/photos/upload/route.ts
export async function POST(request: NextRequest) {
  // 1. Auth check — server Supabase client
  // 2. Parse multipart form data — Next.js built-in (no multer)
  // 3. Validate: max 10MB, accepted types: image/jpeg, image/png, image/webp
  // 4. Upload via admin Supabase client:
  //    supabaseAdmin.storage.from("user-photos").upload(`${userId}/${uuid}.webp`, buffer)
  // 5. Return: { publicUrl: string, storageKey: string }
}
```

No server-side resizing needed — browsers handle this client-side before upload.

### Photo-Slide Association

Photos are not stored in the `generations` table at upload time. They are associated with slides in the carousel JSON when the user assigns them in the UI.

The existing `SlideData.imageUrl` field (already present, used for Photo Mode) will carry the Supabase public URL for user-uploaded photos. No schema change needed.

The generation UI needs:
1. An upload area (separate from the generate flow — before or after generation)
2. A "assign photo to this slide" action per slide in the carousel preview
3. Zustand store to hold `uploadedPhotos: { storageKey: string, publicUrl: string }[]`

When the user assigns a photo to a slide: `slide.imageUrl = photo.publicUrl`. The template already knows how to render a photo background — `PhotoBackground` component already exists in `utils.tsx`.

### Two Photo Layouts Using Existing Infrastructure

`layout: "split"` with `imageUrl` set = photo on one half, text on the other.
`layout: "default"` with `imageUrl` set = photo as full background with text overlay (existing `PhotoBackground` behavior).

No new photo rendering code needed — the template switch handles placement.

### Cleanup Consideration

User photos in Storage will accumulate. Acceptable for MVP — add a cleanup cron job in a later milestone (delete photos older than 30 days that are not referenced in any generation).

## 4. Content Calendar

### Component Boundaries

The content calendar is a self-contained page under the dashboard route group. It does not modify any existing component. It reuses `/api/generate` for individual carousel generation.

```
app/(dashboard)/content-plan/
  page.tsx              — main calendar view, server component for initial data fetch
  ContentPlanClient.tsx — client component (interactive calendar, plan editing)
  GenerateSingleModal.tsx — modal that calls /api/generate for one topic
```

### New Supabase Table

```sql
CREATE TABLE content_plans (
  id           UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  user_id      UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  title        TEXT NOT NULL,                    -- e.g. "March 2026"
  month        TEXT NOT NULL,                    -- "2026-03"
  items        JSONB NOT NULL DEFAULT '[]',      -- ContentPlanItem[]
  created_at   TIMESTAMPTZ DEFAULT now(),
  updated_at   TIMESTAMPTZ DEFAULT now()
);

ALTER TABLE content_plans ENABLE ROW LEVEL SECURITY;
CREATE POLICY "Users own plans" ON content_plans
  FOR ALL TO authenticated
  USING (user_id = auth.uid())
  WITH CHECK (user_id = auth.uid());

CREATE INDEX idx_content_plans_user_month ON content_plans(user_id, month);
```

**ContentPlanItem shape (stored in `items` jsonb column):**
```typescript
interface ContentPlanItem {
  id: string;           // client-generated UUID
  date: string;         // "2026-03-15" ISO date
  topic: string;        // "Как построить личный бренд за 30 дней"
  post_type: "carousel" | "reels-idea" | "story-idea" | "text-post";
  status: "idea" | "generated" | "published";
  generation_id?: string;  // FK to generations.id if carousel was generated
  notes?: string;
}
```

Storing items as JSONB in a single row per month avoids a separate join table and simplifies the update pattern (PATCH the whole items array on any change).

### New API Routes

**`POST /api/content-plan/generate-plan`** — AI generates 30-day plan:

```typescript
// Input: { niche: string, month: string }
// Steps:
// 1. Auth check
// 2. Build Gemini prompt: "Generate a 30-day content plan for Instagram. Niche: {niche}. Month: {month}. Return JSON array of items..."
// 3. Call Gemini — same pattern as /api/generate (no slot claimed, content plans are not counted toward carousel limit)
// 4. Insert into content_plans table
// 5. Return { planId, items }
```

**`PATCH /api/content-plan/[planId]`** — save edited plan:

```typescript
// Input: { items: ContentPlanItem[] }
// Steps:
// 1. Auth check
// 2. Verify user owns the plan
// 3. UPDATE content_plans SET items = $1, updated_at = now() WHERE id = $2 AND user_id = $3
// 4. Return { success: true }
```

### How /api/generate Is Reused

When a user clicks "Generate carousel" for a content plan item, the UI calls the existing `POST /api/generate` with the item's topic as the text input. This consumes a generation slot normally (counts toward the user's monthly limit). The `generation_id` returned is written back to the `ContentPlanItem.generation_id` field and the plan is patched.

No changes needed to `/api/generate` for this feature. The content plan UI is just a structured way to queue generation calls.

### Page Component Structure

```
content-plan/page.tsx (Server Component)
  - Fetch user's content plans via admin client (or server client with RLS)
  - Pass initial data to client component

ContentPlanClient.tsx (Client Component)
  - State: current plan, editing item, loading states
  - Calendar grid: 30 days × post slots
  - "Generate 30-day plan" button → POST /api/content-plan/generate-plan
  - Per-item: edit topic, change date, click "Generate carousel"
  - GenerateSingleModal: small modal wrapping the generate call

GenerateSingleModal.tsx (Client Component)
  - Props: topic, onGenerated(generationId)
  - Calls POST /api/generate with topic
  - Shows slide preview in modal
  - On confirm: calls PATCH /api/content-plan/[planId] to mark item as generated
```

### State Management

Content calendar state is local to the page component — no new Zustand store needed. The existing `useGenerateStore` is not involved. The modal does its own `fetch('/api/generate', ...)` call and handles loading/error state locally.

## Build Order (Dependencies)

The four features have these dependencies:

```
Photo upload
  └── no dependencies — can build first or in parallel with layout

Layout system
  └── requires: schema change to SlideData + prompt changes
  └── must complete before: rich elements (elements interact with layout)

Rich elements
  └── requires: layout system (big-number layout uses stat element)
  └── requires: SVG element components created

Content calendar
  └── requires: /api/generate works (already does)
  └── no dependency on layout or rich elements — can build in parallel
```

**Recommended build sequence:**

1. **Photo upload** — standalone, unblocks "photo as slide content" UI early. Delivers user-visible value immediately.

2. **Layout system** — schema change + prompt update + template implementation. Forms the foundation for rich elements. No UI work beyond template changes — the slide preview already picks up layout automatically.

3. **Rich elements** — builds on layout. Requires element components and template integration. Most complex visually.

4. **Content calendar** — independent of 1-3. Can be built in parallel with rich elements. No shared state or API contract changes.

## Data Flow Summary

### Photo Upload Flow (complete)

```
UI: FileInput → client resize → Blob
  → fetch POST /api/photos/upload (FormData)
    → createServerClient auth check
    → supabaseAdmin.storage.from("user-photos").upload(path, buffer)
    → return { publicUrl }
  → Zustand: add to uploadedPhotos[]
  → UI: thumbnail strip, user clicks "Add to slide N"
    → Zustand: set slides[N].imageUrl = publicUrl
  → SlideRenderer renders with imageUrl present
    → template component calls PhotoBackground(imageUrl)
    → html2canvas captures including the background image (CORS must allow Supabase CDN)
```

**Critical:** Supabase Storage CDN must have CORS headers allowing the deployment origin. html2canvas will fail to capture cross-origin images if CORS is not configured. Add `Access-Control-Allow-Origin: https://swipely.ru` (and localhost) in Supabase Storage CORS settings.

### Content Calendar Flow (complete)

```
User: enters niche → clicks "Generate plan"
  → POST /api/content-plan/generate-plan
    → Gemini returns 30 ContentPlanItem[]
    → INSERT into content_plans
    → return items
  → UI renders calendar grid

User: clicks "Generate carousel" on item N
  → GenerateSingleModal opens with item.topic prefilled
  → User confirms → POST /api/generate (existing endpoint, unchanged)
    → claim_generation_slot (counts toward monthly limit)
    → Gemini returns slides
    → saved to generations table
    → return { slides, post_caption } + generation.id
  → UI shows slide preview in modal
  → User confirms → PATCH /api/content-plan/[planId]
    → items[N].status = "generated", items[N].generation_id = id
```

## Anti-Patterns to Avoid

### Separate template components per layout variant
Creating `SwipelySlideQuote.tsx`, `SwipelySlideNumber.tsx` etc. would mean duplicating the entire decorative layer (glows, grid, decorations, logo, footer) for each layout. Keep layout as a render branch inside one component file.

### Storing photo as base64 in slide JSON / generations table
The existing Photo Mode stores base64 in `imageUrl` because Gemini returns base64. User-uploaded photos must use Supabase Storage public URLs. Storing base64 for user photos would bloat the `generations.output_json` column (potentially several MB per generation).

### Calling /api/generate inside /api/content-plan/generate-plan
Content plan generation is a different Gemini call (returns a plan structure, not slides). These are separate prompts and should be separate route handlers. /api/generate should not be called server-side from another route — it is a client-facing endpoint with auth and slot-claiming logic that assumes a real user session.

### Canvas-based charts (Chart.js, Recharts with Canvas renderer)
html2canvas only captures DOM elements. Canvas elements painted by third-party libraries are captured as blank rectangles. Use only SVG + inline styles.

### New Zustand store for content calendar
The calendar page has self-contained state (current plan, edit mode, loading). Adding a store adds indirection without benefit. Use React `useState` and `useReducer` locally.
